# Penumbra Organizer v0.3.3-beta

## Changes since v0.3.2-beta

- **New: Folder Cleanup.** Fixes the Critical bug where empty, orphaned folders persist in Penumbra's
  UI after re-sorting or rolling back a change, and never clear, even after fully restarting the
  game. Root cause: Penumbra tracks folder structure separately in `mod_filesystem/organization.json`
  and never prunes it on its own. A new "Folder Cleanup" tab lets you review and select orphaned
  folders (empty, with or without customization) to prune. Pruning goes through the same dry run,
  verified backup, Apply, and rollback steps as any other change.
- Safety gating on the new feature: at most 3 folders pruned per Apply by default, the Apply
  confirmation dialog lists exactly which folders will be pruned, and a post-Apply prompt asks you
  to confirm in Penumbra whether the folder(s) disappeared.
- **New: Advanced Cleanup.** An opt-in mode, gated behind an explicit risk-acknowledgment dialog,
  that removes the 3-folder-per-Apply cap for libraries with hundreds or thousands of orphaned
  folders. Resets every time the app restarts; never persisted.
- The Backups tab can now restore `organization.json` on its own, independent of other files, with a
  warning that restoring it can bring back the orphaned-folder problem this release fixes.
- Added `docs/HOW_TO_USE.pdf`, a full usage guide (also bundled in the release zip) covering the
  whole workflow including Folder Cleanup and Advanced Cleanup, plus a manual-backup walkthrough.

## Validation

Real-install validation was done on two live Penumbra installs, not just automated tests:

- A small library (~220 mods): live repro of *new* orphan creation didn't reproduce; this install
  never had an `organization.json` at all, so it couldn't serve as a positive test case.
- A large library (2,000+ mods, matching the scale of the original reports): live repro of new
  orphans still didn't reproduce, but Folder Cleanup correctly detected real pre-existing orphaned
  entries. Using Advanced Cleanup, more than 3 folders were selected and pruned in a single Apply,
  then rolled back successfully, confirming the read/classify/write/rollback path is correct
  against real, organically-grown data, including batches larger than the default cap. Mods and a
  customized (colored) folder were unaffected.

## Known issues

- Live *creation* of new orphaned folders (the original repro: sort one way, switch scheme, reapply)
  has not been reproduced on any install tested so far. The fix has only been validated against
  folders that were already orphaned before this version was installed.
- See [docs/KNOWN_ISSUE_EMPTY_FOLDERS_AFTER_RESORT.md](../KNOWN_ISSUE_EMPTY_FOLDERS_AFTER_RESORT.md)
  for the full root-cause writeup and validation details.
